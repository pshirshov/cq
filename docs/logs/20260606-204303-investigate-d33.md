## Evidence
1. Browser geometry (current committed CSS, 441d46c), window 1280px, dialog inner ~847px:
   every svg BOX is left-aligned (`svgLeft == sectionLeft`, leftOffset=0) — wide ones
   fill 847px, narrow ones sit at intrinsic width. So the BOX is NOT right-aligned.
2. BUT the screenshot shows the CONTENT of the wide diagrams (milestones/tasks/goals)
   shifted right with a large empty gap on the LEFT; narrow diagrams (defects/…) are
   flush-left. `preserveAspectRatio="xMinYMid"` pins content to xMin (left) and can
   only ever leave a gap on the RIGHT — a LEFT gap proves the leftmost node is not at
   viewBox x≈pad.
3. Per-ledger leftmost-node x (real computeStateMachine):
   - milestones w=856 minNodeX=192 (gap 176) leftmost=blocked
   - tasks      w=856 minNodeX=192 (gap 176) leftmost=wip
   - goals      w=1032 minNodeX=192 (gap 176) leftmost=planned
   - defects/hypothesis/questions/decisions/reviews/handoffs minNodeX=16 (flush)
   This maps EXACTLY to D33's user-confirmed census (RIGHT: goals/milestones/tasks;
   LEFT: all others).
4. Hand-computation of `computeDagLayout` longest-path layering on the milestones
   transition graph (fully cyclic: open/postponed/blocked all mutually reachable, all
   →done) yields layers blocked=1, postponed=2, open=3, done=4 — NO node at layer 0.
   x = pad + layer*(nodeWidth+hGap) ⇒ leftmost node at x=192, layer-0 column (176px)
   empty.

## Root cause
`computeDagLayout` (dagLayout.ts) longest-path layering can assign a minimum layer
> 0 when the graph has no true source node (every node has an incoming edge — the
cyclic milestones/tasks/goals lifecycles). The cycle-break returns 0 for back-edges
but does not place any node on layer 0, so all nodes are offset right by
minLayer×(nodeWidth+hGap), leaving empty leading columns in the viewBox.
preserveAspectRatio=xMinYMid then renders the empty leading padding as a left gap.
The "intrinsic-width≥container" correlation in the original D33 analysis was a
confound (the cyclic graphs are also the widest). CSS/preserveAspectRatio were never
the cause — explaining why 47e8ff7 and 441d46c (both CSS-only) did not fix it.

## Suggested fix
Re-base layers so the minimum layer is 0: after `for (const id of nodeIds)
layerOf(id);`, compute `minLayer = min(layer.values())` and subtract it from every
node's layer before grouping/positioning. Generic and correct for BOTH the
state-machine view and the milestone dependency-graph view (a layered DAG should
never reserve empty leading columns). Add a pure unit test on computeDagLayout (and
computeStateMachine over the milestones schema) asserting minNodeX === pad — a
happy-dom-free regression guard. The CSS from 441d46c can stay (it correctly
prevents wide-diagram overflow); the layout fix is independent.
