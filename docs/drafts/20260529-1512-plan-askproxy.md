## Goal

`ask_user_question` works for Codex sessions. The tool handler runs in the
cq-mcp subprocess, calls back over the existing outer-13 internal WS to the cq
server, the server drives the existing browser ask UI + AskBroker flow, and the
answer is proxied back to cq-mcp where the tool resolves. ClaudeBridge stays
in-process and untouched.

## Verified architecture (from intake reading)

1. **internalProtocol.ts** — `InternalWsMessage` is a Zod `discriminatedUnion("type", [...])`
   with only `ledger.changed`. Every variant carries `sourcePid` for loop-guard.
   Receivers (`server/internalWs.ts`, `cq-mcp/internalWs.ts`) drop unknown types
   with a warning and drop `sourcePid === self.pid`.
2. **server/internalWs.ts `InternalWsService`** — holds an UNKEYED `Set<InternalWsSocket>`;
   `broadcast(msg)` sends to every connected internal client; `registerHandler<T>(type, fn)`
   one-per-type, `fn: (msg) => Promise<void>`. Inbound dispatch is in `message()`.
   It does NOT track per-session identity — there is one server, N cq-mcp children
   (one per live Codex session). `broadcast` reaches all of them; the `askId` (and
   `sourcePid` loop-guard) discriminate. There is no per-socket send keyed by session.
3. **cq-mcp/internalWs.ts `InternalWsChannel`** — `connect`, `send(msg)`,
   `registerHandler<T>(type, fn)`, `close()`. `close()` does NOT notify handlers;
   `isClosed()` exists. No reconnection. There is no "on close" callback hook today —
   adding `rejectAll()` on disconnect needs an explicit close listener wired in main.ts
   OR a small addition to the channel. Decision: add an `onClose(cb)` registration to
   `InternalWsChannel` (minimal) so main.ts can reject pending asks; the WebSocket
   already has a `close` listener in the ctor — extend it to fan out to registered
   close callbacks.
4. **askUserQuestion.ts** — `AskQuestion = unknown`; tool input schema
   `{ questions: z.array(z.any()).min(1).max(4) }`. Two-slot race machine in
   `AskBroker`: `pending` (single) + `bufferedReply` (single), keyed implicitly by
   the single active ask. `ask(toolUseId, questions)` returns
   `Promise<{questions, answers: Record<string,string>}>`; `reply(toolUseId, answers)`
   resolves pending or buffers; `rejectAll()` rejects pending + clears buffer.
   `normaliseAnswers` joins string[] with ", ". The MCP tool result shape is
   `{ content: [{ type:"text", text: JSON.stringify(output) }] }` where
   `output = { questions, answers }`.
5. **claudeBridge.ts** — `toolAliases: { AskUserQuestion: "mcp__cq__ask_user_question" }`;
   the in-process `cq` MCP server owns `createAskUserQuestionTool(askBroker)`.
   The question UI is driven by the SDK's NATIVE assistant stream: the SDK emits an
   assistant message whose content has a `tool_use` block `name:"AskUserQuestion"`;
   the browser (Stream.tsx:539) renders the ask card from `toolUse.id` + `toolUse.input`.
   There is NO separate "emit ask to browser" server function today — Claude gets it
   for free from the SDK event stream. `handleChatQuestionReply` just calls
   `askBroker.reply(frame.toolUseId, frame.answers)`.
6. **protocol.ts `ChatQuestionReply`** = `{ type, seq, ts, sessionId, invocationId,
   toolUseId, answers: Record<string,unknown> }`. Browser→server (ChatTab.tsx:710).
7. **bridge.ts facade** — `handleChatQuestionReply(ws, frame)` routes to
   `this.active ?? this.claude`. When a Codex session is active, `this.active === codex`,
   so the reply already lands in `CodexBridge.handleChatQuestionReply` (currently a
   no-op, codexBridge.ts:621). **This is the proxy reply hook.** The brief's "decide by
   session platform" is ALREADY satisfied structurally by `this.active` routing —
   no redundant platform branch needed, and adding one risks conflict. We make the
   dispatch explicit + tested by giving CodexBridge a real `handleChatQuestionReply`.
8. **codexBridge.ts** — spawns cq-mcp via `CodexOptions.config.mcp_servers.cq =
   { command, args, env }`. `env` already carries `CQ_INTERNAL_WS_URL` +
   `CQ_INTERNAL_WS_TOKEN` (lines 396-401). `ActiveCodexSession.ws` holds the live
   browser socket + `chatSessionId` + `invocationId`. No system prompt is set;
   Codex auto-discovers MCP tools via the CLI's tools/list — so NO prompt change is
   needed (document this). `handleChatQuestionReply` is a no-op to replace.

## Browser-render shape (the synthetic frame the proxy must emit)

The browser renders an ask card when it sees a `chat.event` whose `sdkEvent` is an
assistant message with a `tool_use` block named `AskUserQuestion`. CodexBridge
already has `sendEvent(ws, session, msg)` that wraps any object as a `chat.event`.
So the server proxy emits, via `sendEvent`:

```
{
  type: "assistant",
  message: {
    id: <some id>,
    role: "assistant",
    content: [{ type: "tool_use", id: <toolUseId>, name: "AskUserQuestion", input: { questions } }],
    model: <session.model>,
  },
  parent_tool_use_id: null,
}
```

This is the ONE "emit ask to browser" implementation. Factor it into a shared
helper so the shape is defined once (and unit-testable). It lives server-side; the
Claude path does not call it (Claude gets the SDK's native event). The helper is
new code, not a refactor of Claude — so "both paths use one implementation" is
honored to the extent that ONE function builds the synthetic ask event; Claude
simply never needs it because the SDK emits the real one. Document this asymmetry.

## Correlation design (the adversarial-review crux)

Per ask, three identifiers:
- `askId` — generated by cq-mcp, unique per ask. Drives the cq-mcp broker map AND
  the `ask.reply` loop-back discrimination.
- `toolUseId` — generated by cq-mcp too (so it is known at ask.request time and put
  into the synthetic assistant tool_use block). The browser echoes it in
  `chat.question_reply.toolUseId`.
- `sessionId` — the cq chat session; routes the synthetic event to the right browser
  socket and lets the proxy ignore replies for other sessions.

Server-side proxy state (owned by a new `AskProxy` collaborator that CodexBridge
holds): a map `toolUseId → { askId, sessionId, sourcePid }`. Flow:
1. `ask.request {askId, toolUseId, sessionId, questions, sourcePid}` arrives at
   `InternalWsService` → dispatched to a handler registered by server wiring that
   calls `askProxy.onAskRequest(msg)`.
2. `askProxy.onAskRequest`: look up the live socket for `sessionId` (via CodexBridge's
   active session — only the active Codex session can legitimately ask; if sessionId
   doesn't match the active session, log + send an immediate `ask.reply` with an error
   marker so cq-mcp does not hang — see "preempt" below). Record
   `toolUseId → {askId, sessionId, sourcePid}`. Emit the synthetic assistant tool_use
   `chat.event` to that socket.
3. Browser answers → `chat.question_reply {sessionId, toolUseId, answers}` →
   facade routes to `CodexBridge.handleChatQuestionReply` → `askProxy.onQuestionReply(frame)`:
   look up `toolUseId`, find `askId`, `broadcast` an `ask.reply {askId, answers, sourcePid:serverPid}`.
   The cq-mcp whose broker has that `askId` resolves; others ignore (unknown askId).
   Delete the map entry.

**Reply-before-ask race on the SERVER side cannot happen** the way it can on cq-mcp:
the browser cannot send `chat.question_reply` for a `toolUseId` it has not yet seen
rendered, and it only sees the toolUseId after the synthetic event is emitted (step 2).
So server-side, `onQuestionReply` always runs after `onAskRequest`. The two-slot race
is a cq-mcp-side concern (the inbound `ask.reply` could in principle be processed
before the tool handler's `broker.ask(askId)` parks — same timing class as Claude's).
We mirror the two-slot machine on cq-mcp keyed by askId for correctness regardless.

**Preempt / shutdown while pending** (both sides must reject, not hang):
- cq-mcp side: `CqMcpAskBroker.rejectAll()` on channel close → tool handler's awaited
  promise rejects → tool returns an error `CallToolResult` (not a hang). Wire via the
  new `channel.onClose(...)`.
- server side: if CodexBridge shuts down / preempts while a proxy ask is outstanding,
  we cannot reach the (now-gone) browser. We do NOT need to actively notify cq-mcp:
  preempt tears down the Codex session, which closes its cq-mcp's stdin → cq-mcp's WS
  closes → its broker.rejectAll fires. But to be robust we also clear the server-side
  map on shutdown. If a `chat.question_reply` arrives after shutdown (stale), the map
  is empty → log + drop (no broadcast), exactly like Claude's stale-reply guard.

## Risks / open questions

- **Does cq-mcp's `tools/list` need the WS channel up to expose `ask_user_question`?**
  Yes — register the tool only when `channel !== null`. Standalone smoke stays 13 tools.
- **Codex CLI passing `CQ_SESSION_ID` through `mcp_servers.cq.env`** — same mechanism as
  the two existing env vars (verified working in outer-13). Low risk.
- **Synthetic assistant event id collisions / Stream dedupe** — use a stable unique id
  (e.g. `ask-${toolUseId}`); the browser keys ask cards by tool_use id.
- **Two cq-mcp children connected at once** (rare: a preempt in flight). `ask.reply`
  broadcast is askId-scoped; the wrong child ignores an unknown askId. Safe.
