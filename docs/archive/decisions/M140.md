    VERDICT: FEASIBLE-WITH-CAVEATS. The hard constraint — advance an orphan git branch (rooted at the zero/empty commit) WITHOUT switching the working tree to it — is proven correct by an executed throwaway PoC against a /tmp repo (never touched the cq repo).
    
    ## Exact plumbing command sequence tested (no checkout)
    Per ledger write, using an ISOLATED scratch index (GIT_INDEX_FILE -> throwaway path) so the real index / git status is never perturbed:
    1. blob:   blob=$(printf '%s' "$content" | git hash-object -w --stdin)
    2. tree:   GIT_INDEX_FILE=$scratch git update-index --add --cacheinfo 100644 $blob docs/tasks.md ; tree=$(GIT_INDEX_FILE=$scratch git write-tree)
    3. commit: commit=$(git commit-tree $tree -m msg)            # ORPHAN (no -p) on first write
               commit=$(git commit-tree $tree -p $parent -m msg)  # child thereafter
    4. ref:    git update-ref refs/heads/cq-ledger $commit        # advance — NO checkout, NO working-tree mutation
    Reads need NO checkout: git cat-file -p cq-ledger:docs/<ledger>.md / git show <sha>:<path>; enumerate via git ls-tree -r --name-only cq-ledger; history is the orphan ref's real commit log (free audit trail). git update-index --cacheinfo into a scratch index is preferred over raw git mktree because it builds nested subtrees (docs/archive/<ledger>/<id>.md) automatically.

    ## Concurrency / locking assessment (vs FsLedgerStore lockfile model)
    Current model: per-ledger AsyncMutex + advisory docs/.locks/<ledger>.lock (O_CREAT|O_EXCL, stale-PID reclaim, bounded LedgerBusyError @5s); global __milestones__ lock; multi-lock order __milestones__ then per-ledger alphabetic. atomicWrite rename per file.
    Findings: (1) update-ref is NOT atomic against a concurrent read-modify-write (read parent tree -> mutate -> commit-tree -> update-ref); two writers off the same parent lose one update (last-writer-wins). The EXISTING AsyncMutex+lockfile already serialises the in-process store vs the cq-mcp child on one cwd (LOCK-D01), so it suffices IF the read-current-tree step runs INSIDE the held lock. (2) Use CAS as defence in depth: git update-ref refs/heads/cq-ledger <new> <expected-old> fails if the ref moved — an optimistic-concurrency check the pure-FS atomicWrite cannot offer; a lost update surfaces as an error instead of silent loss. (3) The advisory .lock files MUST NOT be committed to the orphan ref (pid-scoped, ephemeral) — they stay on the real FS, gitignored, unchanged. So the lock model is UNCHANGED; only the payload write changes from atomicWrite(file) to commit-tree+CAS-update-ref under the same lock.
    
    ## Interaction with the existing per-merge/per-archive cq git-commit steps
    Today cq commands commit the ledger ON THE WORKING BRANCH (implement/advance.md ~L409/~L510 and sibling plan/investigate commands: `git add docs/ ; git commit -q -m "chore(ledger): …"`). An orphan-ref ledger REPLACES these: stop tracking docs/*.md on the working branch (gitignore), and DROP the command-level chore(ledger) commit steps entirely because the FsLedgerStore write path already advanced the orphan ref per mutation (continuous commit). Principal UPSIDE: the ledger stops interleaving chore(ledger) commits into every feature branch / merge, eliminating docs/*.md ledger↔code merge conflicts. New operational surface: the orphan ref must be PUSHED/FETCHED EXPLICITLY (git push origin cq-ledger; not part of a default push of the feature branch); cq commands / CI need an explicit refspec.
    
    ## reflog / push-fetch / failure modes
    - reflog: update-ref writes a reflog for refs/heads/cq-ledger (crash-recovery/undo). Loose objects from hash-object/commit-tree are unreferenced until update-ref lands; a crash between leaves GC-able garbage, no corruption.
    - push/fetch: normal branch ref; force-push hazard from last-writer-wins history rewrites -> recommend non-fast-forward protection + CAS update-ref.
    - clone: a single-branch/shallow clone of the code branch will NOT carry the ledger; tooling must fetch refs/heads/cq-ledger explicitly.
    - linked-worktree FALLBACK: `git worktree add ../ledger-wt cq-ledger` checks out the orphan branch in a SEPARATE dir without touching the main checkout, letting the store use ordinary fs.readFile/atomicWrite + `git -C ../ledger-wt commit` per write; simpler (no plumbing) but adds a second working tree + lock domain. Pure-plumbing path is recommended primary; linked worktree is the documented fallback.
