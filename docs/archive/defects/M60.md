- headline: Web BatchAnswerModal dismisses prematurely after answering a non-last question (should advance to next)
- severity: medium
- description: |
    Web UI regression: in the Q&A batch-answer popup (BatchAnswerModal in @cq/ledger-web), clicking "save & mark answered" closes/dismisses the entire popup even when the just-answered question is NOT the last one in the batch — it should instead advance to the next unanswered question and only close when the batch is exhausted. The "as recommended" button may have the same problem (unconfirmed). Suspected regression from the recent D29 fix (task T163), which modified the BatchAnswerModal submit/disable behavior and adjusted batchModalClose tests. Expected behavior: answering a non-terminal question advances within the modal; answering the last one closes it.
    
    Severity rationale: degraded UX (the user must reopen the popup for each remaining question) but a workaround exists (reopen), so `medium`.
- rootCause: "CONFIRMED (H24, user repro + validated source). The premature close is NOT in batchSave (the prior round correctly cleared that) — it is the modal's BACKDROP click-to-dismiss. The batch overlay is `<div className=\"lw-help-backdrop\" data-testid=\"batch-overlay\" onClick={onClose}>` (ledger-web/src/App.tsx) with the inner dialog stopping click propagation; onClose === setBatchOpen(false). It closes on ANY click whose common-ancestor is the backdrop, with NO guard that the press STARTED on the backdrop. Mechanism: the 'save & mark answered' HoldButton fires onConfirm on a HOLD_MS TIMER (HoldButton.tsx ~115), not on pointerup — so while the user is STILL pressing, batchSave advances batchIndex to the next (shorter) question and the content-driven `.lw-batch` (no fixed height, only max-height:90vh; backdrop is a centering flexbox) shrinks; the still-held pointer ends up over the backdrop; on release the browser synthesizes a click whose common ancestor is the backdrop → onClose fires → the modal dismisses mid-queue. HoldButton never stopPropagation()s the release, so nothing suppresses it. This is the classic press-started-inside / released-outside modal-dismiss bug (react-modal #466). The test suite was GREEN because batchModalClose.test.tsx never simulates a backdrop click (its holdFull helper dispatches only pointerdown + clock advance, no pointerup/click) — a vacuous-coverage gap (cf. D24/H14). The SAME vulnerable `onClick={onClose}` backdrop pattern exists in two other overlays: HelpOverlay (App.tsx ~1485) and the log modal (~3021)."
- ledgerRefs: ["goals:G21"]
- suggestedFix: "Gate backdrop dismissal on the press having STARTED on the backdrop: track onMouseDown/onPointerDown on the backdrop (record whether target === currentTarget, i.e. the backdrop itself) and in the close handler only call onClose() when BOTH the recorded down-target AND the up/click target are the backdrop. Extract a small shared safe-backdrop wrapper (or hook) and apply it to ALL THREE overlays (batch + help + log) so the whole class is fixed, not just the reported one. REPRODUCE-FIRST (happy-dom, testable): a test that opens the batch modal, dispatches pointerdown INSIDE the dialog (on the submit/hold button) then a `click` on data-testid=\"batch-overlay\" (the backdrop), and asserts the modal STAYS OPEN — this FAILS today (modal closes). Keep the existing queue-drain close test green (an explicit backdrop click that both starts AND ends on the backdrop must still close). A11y/Escape-to-close behaviour unchanged. Doc-confirm with bun run check + the new failing→passing test."
