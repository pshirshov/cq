## Confirmed defect — D-UNIFASYNC-01 (fixed)

`packages/server/src/agent/codexBridge.ts:94` declared:

```ts
export type CodexFactory = (options?: CodexOptions) => Codex | Promise<Codex>;
```

The production factory (constructor lines 289–298) is already async (dynamic `import("@openai/codex-sdk")`), and the only call site (`await this.codexFactory(codexOptions)` at line 382) already awaits. The union existed solely so three test factories could return a `Codex` synchronously. This violates the canonical rule from `feedback-uniform-async`: lazily-initialised singletons / memoised factories should be uniformly `Promise<T>`, not `T | Promise<T>` unions.

**Fix.** Tightened the type to `=> Promise<Codex>` and updated the three test factories:

- `packages/server/test/codexBridge-mcp.test.ts:49` and `:98`
- `packages/server/test/codexBridge.test.ts:120`

All became `async (...): Promise<Codex> => …`. No call-site changes required (the call site already awaited).

### Smell 3 — Lazy-init races

`FsLedgerStore.init()` (`packages/ledger/src/store/FsLedgerStore.ts:101`) uses a `private initialised = false` flag rather than memoising an in-flight `Promise<void>`. Two concurrent `init()` calls *could* race past the guard.

Audit: `init()` is called exactly once per process in the surveyed entry points (`packages/server/src/server.ts:47`, `packages/server/src/devServer.ts:46`, `packages/cq-mcp/src/main.ts:359`) — each is a strictly serial startup step that awaits before any tool registration. No concurrent caller exists.

**Not-a-smell — documented.** The boolean guard would become a race if a second concurrent caller were added; the current code is correct for the existing call-graph. If future code adds a parallel init caller, switch to the canonical memoised-promise pattern. Not in scope to refactor now (no concrete race, and it predates this diff).

`Bridge.codex: CodexBridge | null = null` (`packages/server/src/agent/bridge.ts:99`) is lazily constructed in `resolveBackend` — sync construction, not async. Not a sync/async union, not a lazy-init race. **Not a smell.**

### Smell 5 — Eager construction that should be lazy

`CodexBridge` constructor calls `defaultResolveCqMcpBin()` (walks parent dirs, `fs.statSync`) and `defaultDetectCodexAuth()` is set as a lambda (not invoked).

- `defaultResolveCqMcpBin` runs once per `CodexBridge` instance (not per chat.start). `Bridge.codex` is itself lazy (constructed on first Codex `chat.start`), so the resolver only fires when a Codex session actually starts. The result is cached on `this.cqMcpBin` and consumed by every subsequent `chat.start`. Paying the resolution cost once at first-Codex-start matches the brief's "default cost should be paid lazily on first use" — `Bridge.resolveBackend` IS the lazy gate.
- `defaultDetectCodexAuth()` is wrapped in a lambda (`opts.detectAuth ?? ((): boolean => defaultDetectCodexAuth())`); the actual `fs.statSync(authPath)` only runs inside `handleChatStart`, not at construction. Fine.

**Not a smell — documented.** Construction does NOT perform synchronous I/O that could fail; the lambdas defer the I/O. The bin resolver IS sync I/O but is lazy via the Codex-bridge laziness.

### Smell 6 — Dispose paths that don't drain in-flight work

`CodexBridge.shutdown()` (lines 620–641) calls `session.abortController?.abort()` and finalises persistence rows synchronously, but does NOT await the in-flight `handleChatInput`'s `for await` loop. The aborted controller will propagate into the codex-sdk stream, but the bridge does not hold a handle to the input promise.

Audit: this matches the pre-existing `ClaudeBridge.shutdown()` shape exactly (same abort-and-finalise pattern). It predates the outer-7..outer-10 diff. The brief scoped this smell to "any *new* lazily-initialised promise field". No such field was added — the per-session controller has always existed. D-LED-06 (already closed) covered the analogous `FsLedgerStore.dispose()` drain; that fix stands.

**Not a smell — documented.** Pre-existing pattern, no new field, no concrete regression observed. Aligning the codex/claude shutdown drain semantics is a separate cross-cutting concern, not a uniform-async violation.

### Smell 7 — `Promise.race` for "first one wins" without losing-side cleanup

`grep -rn "Promise.race" packages/server/src/ packages/cq-mcp/src/ packages/ledger/src/ packages/web/src/`: one match.

- `packages/server/src/shutdown.ts:54` — `Promise.race([waitForIdle, sleep(timeoutMs)])`. The "loser" is the bare `sleep(timeoutMs)` timer (just a setTimeout / setInterval wrapper) or the `waitForIdle` poll. Neither holds an external resource. The timer fires unconditionally; the idle poll terminates on its own when the bridge clears. Predates the outer-7..outer-10 diff. **Not a smell.**

### Smell 8 — Synchronous I/O in hot paths

`grep -rn "statSync\\|readFileSync" packages/server/src/agent/ packages/cq-mcp/src/`: four matches, all in `codexBridge.ts`:

- `:173` — `whichOnPath()` POSIX `which` implementation. Called only by `defaultResolveCqMcpBin()`, which runs once at CodexBridge construction. Not hot.
- `:225, :240` — `defaultResolveCqMcpBin()` walking node_modules and packages tree. Same — once at construction.
- `:268` — `defaultDetectCodexAuth()` checks `~/.codex/auth.json`. Called once per `handleChatStart` (one per session start, not per turn or per frame). Not hot.

**Not a smell — documented.** All four `statSync` calls are bounded to startup or per-session boundaries, not per-frame / per-turn hot paths. Converting them to async would force the constructor itself to become async (anti-pattern) and complicate `handleChatStart` for no observable benefit.
