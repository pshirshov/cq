### Parser choice (Q5 verification)

Per `CLAUDE.md` § 2 ("recent stable versions") and § 9 ("RTFM"), the
ecosystem with the largest mindshare for markdown round-tripping is
unified/remark, and it is what the brief recommends. `remark-stringify`
preserves AST round-trips for the structural subset we control
(headings, paragraphs, lists, code blocks, yaml frontmatter). We design
the **on-disk schema** to use only the structural subset that round-trips
cleanly; any human edit that diverges (e.g. unusual emphasis nesting)
will normalize on the next write, which is acceptable behaviour for a
machine-authored ledger.

Final stack: `unified@^11`, `remark-parse@^11`, `remark-stringify@^11`,
`remark-frontmatter@^5`, `yaml@^2` (for frontmatter and ledgers.yaml).

## 2. On-disk format (spec)

### `./docs/ledgers.yaml` (central registry)

```yaml
version: 1
ledgers:
  - name: defects
    schema:
      statusValues: [open, in-progress, blocked, resolved, abandoned]
      terminalStatuses: [resolved, abandoned]
      fields:
        severity:    { type: string,   required: true }
        location:    { type: string,   required: true }
        description: { type: string,   required: true }
        rootCause:   { type: string,   required: false }
        fix:         { type: string,   required: false }
        relatedIds:  { type: 'id[]',   required: false }
  - name: tasks
    schema:
      statusValues: [open, in-progress, blocked, done]
      terminalStatuses: [done]
      fields:
        notes: { type: string, required: false }
```

### `./docs/<ledger>.md` (active ledger file)

```markdown
---
ledger: defects
counters:
  milestone: 3
  item: 27
archives:
  - id: M1
    path: ./archive/defects/M1.md
    summary: "M1 — initial spike (12 items, all resolved)"
  - id: M2
    path: ./archive/defects/M2.md
    summary: "M2 — schema rewrite (8 items)"
---

# defects

## M3 — current milestone title

Optional description paragraph.

### D25 — open

- severity: minor
- location: packages/web/src/Stream.tsx
- description: |
    Multi-line field uses YAML block scalar.

### D26 — resolved

- severity: nit
- location: Header.tsx
- description: short description
- rootCause: explanation
- fix: what changed
```

**Round-trip rules:**

- Frontmatter is YAML; `archives` is an ordered array of pointers.
- Counters are a YAML map `{milestone: N, item: M}`; the library
  increments and persists atomically with each `create_*` call.
- Each milestone starts with `## <milestone-id> — <title>` (em-dash);
  an immediate following paragraph (if any) is the description.
- Items are `### <item-id> — <status>` followed by a YAML-shaped
  bullet list of `key: value` pairs. Multi-line values use YAML block
  scalars (`|`). `id[]` values render as inline JSON arrays.
- Unknown sections (anything outside a recognized header) is preserved
  verbatim under a `comments` field on the nearest parent so that
  hand-written notes survive round-trips. (v1: tolerate but warn.)

### `./docs/archive/<ledger>/<milestone-id>.md`

Same body schema as the active file but with no frontmatter
(`archives` list is meaningless inside an archive). The archive
contains exactly one milestone block plus all its items.

## 4. Public TypeScript shape

```ts
// types.ts
export type FieldType = 'string' | 'string[]' | 'id' | 'id[]' | 'timestamp';
export interface FieldSpec { type: FieldType; required: boolean; }
export interface LedgerSchema {
  statusValues: string[];
  terminalStatuses: string[]; // subset of statusValues
  fields: Record<string, FieldSpec>;
}
export type FieldValue = string | string[] | number; // number for timestamp

export interface Item {
  id: string;
  milestoneId: string;
  status: string;
  fields: Record<string, FieldValue>;
  createdAt: number;
  updatedAt: number;
}
export interface Milestone {
  id: string;
  title: string;
  description: string;
  items: Item[];
}
export interface ArchivePointer {
  id: string;          // milestone id
  path: string;        // relative to ledger root
  summary: string;
}
export interface Ledger {
  id: string;
  schema: LedgerSchema;
  counters: { milestone: number; item: number };
  milestones: Milestone[];
  archivePointers: ArchivePointer[];
}

// store/LedgerStore.ts
export interface LedgerStore {
  init(): Promise<void>;       // load registry + all ledger files
  enumerate(): string[];
  fetch(ledgerId: string): Ledger;                     // returns active items + archive pointers
  fetchArchive(ledgerId: string, archiveId: string): Milestone;
  fetchMilestone(ledgerId: string, milestoneId: string): Milestone;
  updateMilestone(ledgerId: string, milestoneId: string, patch: { title?: string; description?: string }): Promise<Milestone>;
  fetchItem(ledgerId: string, itemId: string): Item;
  updateItem(ledgerId: string, itemId: string, patch: Partial<Pick<Item,'status'>> & { fields?: Record<string, FieldValue> }): Promise<Item>;
  createItem(ledgerId: string, milestoneId: string, init: { id?: string; status: string; fields: Record<string, FieldValue> }): Promise<Item>;
  createMilestone(ledgerId: string, init: { id?: string; title: string; description?: string }): Promise<Milestone>;
  createLedger(name: string, schema: LedgerSchema): Promise<void>;
  archiveMilestone(ledgerId: string, milestoneId: string, summary: string): Promise<ArchivePointer>;
  search(ledgerId: string, query: string): Item[];
  dispose(): Promise<void>;
}
```

## 5. Locking and concurrency contract

- Per-`ledgerId` async mutex (a `Promise<void>`-chained queue).
- On any mutating call:
  1. Acquire the in-process mutex for the ledger.
  2. Open `./docs/.locks/<ledger>.lock` exclusively (`O_CREAT|O_EXCL`).
     If file exists, read it: `{pid, hostname, startedAt}`. If `pid`
     is not alive in this process namespace (probed by
     `process.kill(pid, 0)`), unlink and retry once. If still alive,
     throw `LedgerBusyError`.
  3. Mutate the in-memory `Ledger` object.
  4. Serialize to `./docs/<ledger>.md.tmp`, `fs.fsync`, atomic
     `rename` to `./docs/<ledger>.md`.
  5. Unlink lockfile.
  6. Release the mutex.

The `.lock` file content is the JSON `{pid, hostname, startedAt}` of
the current process; `startedAt` lets the reclaim logic distinguish
"same PID was recycled" from "same PID is genuinely us".

**Concurrent calls to different ledgers** run in parallel (separate
mutexes, separate lockfiles).

**Read paths** read from in-memory state, no lock acquired.

## 6. MCP tool surface

Tools registered on the existing `cq` MCP server (extend
`createAskUserQuestionMcpServer` to also include ledger tools — or
better, add a *sibling* factory `createLedgerMcpTools(store)`
returning `tool[]` that is concatenated into the same `cq` server in
`bridge.ts`; this preserves separation of concerns).

Decision: **sibling factory**, concatenated into the `cq` server.
Rationale: `askUserQuestion.ts` should remain focused; tests stay
narrow.

Tool names (auto-allowed by existing `mcp__cq__*` rule in `canUseTool`):

| Tool name | Wire signature |
|---|---|
| `mcp__cq__enumerate_ledgers` | `() → { ledgers: string[] }` |
| `mcp__cq__fetch_ledger` | `(ledger_id) → { schema, milestones:[{id,title,description,items:Item[]}], archivePointers:[…] }` |
| `mcp__cq__fetch_ledger_archive` | `(ledger_id, archive_id) → { milestone }` |
| `mcp__cq__fetch_milestone` | `(ledger_id, milestone_id) → { milestone }` |
| `mcp__cq__update_milestone` | `(ledger_id, milestone_id, title?, description?) → { milestone }` |
| `mcp__cq__ledger_fetch` | `(ledger_id, item_id) → { item }` |
| `mcp__cq__ledger_update` | `(ledger_id, item_id, status?, fields?) → { item }` |
| `mcp__cq__create_item` | `(ledger_id, milestone_id, status, fields, id?) → { item }` |
| `mcp__cq__create_milestone` | `(ledger_id, title, description?, id?) → { milestone }` |
| `mcp__cq__create_ledger` | `(name, schema:{statusValues,terminalStatuses,fields}) → { ledger }` |
| `mcp__cq__archive_milestone` | `(ledger_id, milestone_id, summary) → { pointer }` |
| `mcp__cq__search_items` | `(ledger_id, query) → { items: Item[] }` |

All errors are thrown from the handler; the SDK converts them to a
tool error result. We surface enough diagnostic detail in the message
for the agent to self-correct (e.g. "milestone M3 has 2 non-terminal
items: D24, D27").

## 10. Risks

- **R1: remark round-trip not lossless for our schema.** Mitigation:
  T1 is property-style; design schema to use only structural subset
  remark guarantees (no inline emphasis, no nested lists, etc.).
- **R2: Bun fs.rename atomicity across filesystems.** Mitigation:
  `.tmp` + `rename` is POSIX-atomic on same fs; tmp file lives in
  same `./docs/` dir.
- **R3: SDK-MCP tool input schemas drift.** Mitigation: mirror the
  pattern from `askUserQuestion.ts` (loose `z.any()`-permissive
  where needed for subprocess compat).
- **R4: Lockfile stale-PID detection race.** Mitigation: use
  `O_CREAT|O_EXCL`, retry exactly once after reclaim, document
  that a third party concurrently racing for reclaim wins (we are
  the only writer in this server-on-a-VPN deployment).
- **R5: Resume-from-history parallel task collision.** Mitigation:
  scope strictly excludes `packages/web/src/history/`, ResumePicker,
  Header.tsx, session.title schema, Haiku service. We touch
  bridge.ts and persist/ only enough to inject `ledgerStore`; no
  edits to session schema or invocation rows.
- **R6: codegraph index lag.** Not a blocker; the index is read-only
  guidance.
