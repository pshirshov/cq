- headline: Ledger accepts empty/whitespace-only answer when marking a question `answered`
- severity: medium
- description: |
    The ledger accepts an empty (or whitespace-only) `answer` when a `questions` item is transitioned to `answered` — i.e. a question can be marked answered with no actual answer content. Empty/whitespace-only answers should be rejected at the boundary (the MCP `update_item` answer transition, and correspondingly the TUI and web answer inputs should disable/refuse empty submission). This blocks the flow invariant that an `answered` question carries a usable answer for the planner/orchestrator to consume.
    
    Severity rationale: degraded behaviour (an `answered` question can carry no usable answer, breaking the planner/orchestrator's consume-the-answer contract) but a trivial workaround exists (provide a non-empty answer), so `medium`.
- rootCause: |
    Confirmed (H19 backend + H20 frontend). The ledger enforces NO non-empty constraint on a question's `answer` when it transitions to `answered`.
    
    BACKEND (authoritative): QUESTIONS_SCHEMA.answer is `{type:'string', required:false}` with no content/status-conditional constraint (constants.ts:217); the LedgerSchema FieldSpec type carries only `type`+`required`, so a non-empty constraint cannot be expressed declaratively today. The store transition path applyUpdateItem (core.ts:259-291) runs assertTransitionAllowed + an OPTIONAL StatusChangePrecondition + validateFields; validateFields/assertFieldType only type-check (core.ts:831-859), and the precondition hook is wired ONLY for GOALS_LEDGER (FsLedgerStore.ts:571-581) — so no answer-content check runs for questions. The MCP update_item handler forwards status+fields verbatim (ledgerTools.ts:309-317) and its Zod fieldsSchema accepts '' (ledgerTools.ts:171-173).
    
    FRONTEND (complementary): all four submit paths — web detail submitAnswer/HoldButton (App.tsx:2611/2629), web BatchAnswerModal submit (App.tsx:1732), TUI answer overlay via TextPrompt Enter (TextPrompt.tsx:30-43), TUI BatchAnswerOverlay Enter (app.tsx:1952) — are unguarded and forward an empty/whitespace answer into update_item(status:answered). The existing `answerHasText` guards only prevent clobbering a draft on the 'as recommended'/'pick' buttons; they do NOT block empty submission.
- suggestedFix: |
    Enforce the guard at the AUTHORITATIVE MCP/store boundary (frontends are pure MCP clients per CLAUDE.md), with a complementary UI affordance:
    
    1. BACKEND (primary, the invariant): reject transitioning a `questions` item to `answered` unless the resulting `answer` is present and non-whitespace. Implement as a questions-specific StatusChangePrecondition mirroring assertGoalPhasePreconditions — wire a precondition for QUESTIONS_LEDGER in FsLedgerStore.updateItem (and the in-memory store) that, on (from != answered) -> (to == answered), asserts the EFFECTIVE answer = (patch.fields?.answer ?? item.fields.answer) is a non-empty trimmed string, throwing a SchemaValidation/precondition error otherwise. Must account for the answer arriving in patch.fields vs already present on the item. Reproduce-first: a failing test asserting update_item(Q, status:'answered', fields:{answer:''}) and {answer:'   '} throw, while a non-empty answer still succeeds; run against BOTH FsLedgerStore and InMemoryLedgerStore (dual-tests).
    
    2. FRONTEND (UX): disable the 'save & mark answered' submit when the trimmed answer is empty — web detail HoldButton (App.tsx:2629), web BatchAnswerModal (App.tsx:1732), TUI TextPrompt Enter / BatchAnswerOverlay Enter — reusing the existing answerHasText/setAnswerHasText pattern. Prevents a confusing server-rejected submission.
    
    Planning may weigh the targeted precondition (surgical, low-risk — recommended) vs a general FieldSpec extension (e.g. nonEmpty/requiredOnStatus).
