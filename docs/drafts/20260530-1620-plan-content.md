# Cycle: pcontent — `/plan` planning CONTENT visible + explore-once

Base `2bb2c4b`, worktree `.claude/worktrees/planning-content`, branch `planning-content`.
Baseline: `bun test` 1109/0; e2e 28/28.

## Problem (live dogfood)

1. **Empty Detail.** WF-HIST records a `/plan` run as a workflow session + root
   invocation + one child per phase, but writes **no event rows**. History Detail
   (`Detail.tsx`) replays an invocation's EVENTS via `history.get{replay:true}` →
   `<Stream mode="replay">`; a chat persists events via
   `persistence.events.append(invocationId, msg)`. The workflow records none → the
   Detail of a planning run is EMPTY.
2. **Q&A not visible.** The user answers in the Goals tab (`question.answer` →
   ledger) but the planning transcript has no record of the asked/answered Q&A.
3. **Explore-once.** Every phase subagent re-explores the repo from scratch
   (PLAN-D01's explore-first), so producer → clarify-reviewer → planner each pay
   full cost+latency (the cause of PHASE-TIMEOUT-01's 300s bump).

## Design

### A. Record phase event streams (fixes empty Detail)

- `WorkflowHistoryRecorder` gains:
  - `appendPhaseEvent(phase: PhaseInvocationHandle, msg: SDKMessage)` — Persistent
    → `persistence.events.append(phase.invocationId, msg)`; Null → no-op.
  - `closePhaseEvents(phase)` — Persistent → `persistence.events.close(…)`; Null no-op.
  - `appendRootEvent(handle: WorkflowRunHandle, msg)` — under the root invocation
    (for the Q&A transcript, change B).
- `ProduceRequest` + `PhaseRequest` gain optional `onEvent?: (msg: SDKMessage) => void`.
- Both Claude lanes (`claudeProducer.produce`, `claudePhaseSubagent.dispatch`)
  already iterate the SDK message stream in their `drain` loop (to extract usage);
  forward EVERY drained `msg` to `req.onEvent` there — no second iteration. This
  captures the assistant reasoning/text, the `submit_*` tool_use, and the `result`.
  (The drain currently breaks on the first `result`; the `result` IS forwarded
  before the break, so reasoning + tool_use + result are all captured.)
- `WorkflowRuntime.recordPhase` wires `onEvent: (m) => history.appendPhaseEvent(phase, m)`
  into the dispatch request, and calls `history.closePhaseEvents(phase)` on
  settle (both completed + failed paths). The producer dispatch in `startPlan`
  (which does not go through `recordPhase`) wires the same onto `producerPhase`.
- Codex (`codexHeadless`): forward whatever thread events the lane exposes (coarser;
  document). The Claude lane is the one exercised by tests + e2e this cycle.

### B. Q&A transcript (fixes answers-not-visible)

- On every question-batch write (`writeArtifacts`, `writeIncrement`, and the
  clarify/review new-question writes) the harness appends a synthetic
  assistant-style SDKMessage under the run root listing the questions:
  `{type:"assistant", message:{role:"assistant", content:[{type:"text", text:"Asked N clarifying question(s):\n- …"}]}}`.
- In `submitAnswer`, on the open→answered transition, append a synthetic
  user-style SDKMessage under the root: `Answered <Qid>: <answer>`.
- Both reuse the SDK-message event shape (same as `claudeBridge` echo) so
  `<Stream mode="replay">` renders them as bubbles. Root events interleave with
  the per-phase children in the Detail of the ROOT invocation; the per-phase
  Detail shows that phase's transcript. (Detail opens per-invocation; the History
  list lets the user open the root `main` row to see Q&A + the producer's output
  region, or a child row for a single phase.)

### C. Explore-once

- `grounding` added to `ProducerOutputSchema` (`z.string().optional()` — a producer
  that does not ground still validates; harness defaults to "") AND
  `claudeProducer.submitPlanSchema` (lockstep — GOAL-TITLE-01) AND `GOALS_SCHEMA`
  (`grounding: { type: "string", required: false }`).
- Producer prompt: ask for a concise (few-hundred-word) `grounding` summary —
  what cq is, the ledgers, the relevant subsystems — AFTER exploring.
- Harness persists `grounding` on the goal (`writeArtifacts`: write
  `fields.grounding = output.grounding ?? ""`; continuation refreshes it if the
  producer returns a non-empty one).
- Later-phase prompt builders (`buildClarifyReviewPrompt`, `buildPlannerPrompt`,
  `buildContinuationPlannerPrompt`, `buildPlanReviewPrompt`) take an optional
  `grounding` arg, PREPEND it as a `## Project grounding (already explored — do
  targeted reads only if needed; do NOT re-explore from scratch)` block, and the
  explore instruction is SOFTENED for those phases. The producer keeps the full
  `EXPLORE_FIRST_INSTRUCTION`. `WorkflowRuntime.groundingFor(goalId)` reads
  `goal.fields.grounding`; threaded into every later-phase prompt call.
- Storage justification: GOALS_SCHEMA field vs workflow_session table. The goal is
  the durable record every phase already re-reads (`store.fetchItem(GOALS_LEDGER…)`),
  grounding is goal-scoped, the harness already writes goal fields. The
  workflow_session alternative needs a SQL migration + dual-adapter changes + a
  shared-type change AND couples prompt-building to persistence (the runtime reaches
  persistence only through the recorder). Goal field is lower-coupling /
  lower-complexity; the only cost is the divergence guard → `cq reset` for existing
  dev `docs/goals.md` (gitignored, regenerable; the brief anticipates this).

## Risks / adversarial pre-mortem

- **R1 (GOAL-TITLE-01 repeat):** if `grounding` is in `ProducerOutputSchema` but
  not `submitPlanSchema`, the SDK strips it → strict re-validation could fail →
  producer retries → timeout. Mitigation: add to BOTH; `grounding` is OPTIONAL so
  even a strip degrades to "" rather than a validation failure. Real-SDK
  MockAnthropic test asserts a producer that returns grounding does not time out.
- **R2 (event sink gating control flow):** the drain that forwards events must NOT
  change when `produce`/`dispatch` resolves (still submit-time) nor block. onEvent
  is called synchronously inside the existing drain loop; appendPhaseEvent is a
  sync persistence write (events.append is sync). Best-effort: wrap in nothing that
  throws into the drain (events.append does not throw on the happy path; if it did,
  the existing drain's try/catch already rejects — acceptable, same as usage).
- **R3 (close ordering):** `closePhaseEvents` must run AFTER the last event is
  appended. The drain appends events as they arrive; `settlePhase` runs after the
  dispatch resolves (submit-time), but the `result` event arrives AFTER submit in
  the deferred drain. So closing at settle could close before the `result` is
  appended. Mitigation: the events.append for a closed invocation re-opens the fd
  (SqliteEventLog.fdFor opens lazily on append), so a late append after close still
  lands; AND InMemory append is unconditional. The close is a best-effort fsync,
  not a hard gate. Acceptable; alternatively close in the teardown awaitable after
  drain. → DECISION: close in `closePhaseEvents` called from settle is fine because
  SqliteEventLog reopens on append; but to be correct, also tolerate appends after
  close. Verified by reading events.ts (fdFor reopens). Document.
- **R4 (Detail per-invocation):** Detail opens ONE invocation. The Q&A is under the
  root; phase content is under children. The e2e asserts the Detail of the relevant
  invocation is non-empty. The List already renders root + child rows (WF-HIST-5),
  so the user can open either. The e2e opens the root (Q&A) and a child (phase
  content) — or asserts both via the row the spec targets.
- **R5 (divergence guard):** GOALS_SCHEMA change breaks existing on-disk goals.md.
  Note `cq reset`. No such file in this fresh worktree (verify).
