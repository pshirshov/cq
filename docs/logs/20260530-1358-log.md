## Surprises / constraints for future work

- The SDK `result` message (carrying cost/tokens) arrives only AFTER a SECOND
  HTTP round-trip past the tool submit (tool_result re-enters → model replies →
  end_turn → result). The original drain `break`-on-submit + eager `q.close()`
  cut that off, so usage was never observed. Fix: defer `q.close()` until the
  drain breaks on `result`, bounded by a 5s grace (`closeAfterDrain`); the
  dispatch still resolves at submit-time so pool=1 timing is unchanged. Any
  future change to the headless dispatch teardown must preserve this ordering or
  usage capture regresses to 0.
- The History "nesting" of subagents is in `List.tsx` (flat rows surfaced by the
  UNION CTE + the `↪` marker), NOT `Detail.tsx` (which replays one invocation's
  events). The plan's initial "Detail nests children" framing was corrected in
  adversarial review.
- The goalId→sessionId link lives in a dedicated `workflow_session` table, NOT a
  goal-ledger field — so the divergence-guard / on-disk ledger schema is
  untouched (no `rm docs/*.md` hazard).
