## Design (as built)

- The cq-mcp `ask_user_question` handler mints `askId` (`ask-<pid>-<n>`) +
  `toolUseId` (`<askId>-tu`), sends `ask.request{askId,toolUseId,sessionId,
  questions,sourcePid}`, awaits `CqMcpAskBroker.ask(askId)`, returns the
  Claude-identical `CallToolResult`.
- cq-server's `InternalWsService` `ask.request` handler → `Bridge.handleAskRequest`
  → `CodexBridge.handleAskRequest` → `AskProxy.onAskRequest`: resolves the live
  browser socket for `sessionId` from the active Codex session, emits the
  synthetic `AskUserQuestion` assistant tool_use via `CodexBridge.sendEvent`
  (seq + replay-buffer). If the session is not active it replies-empty so the
  cq-mcp tool does not hang.
- Browser answers → `chat.question_reply{sessionId,toolUseId,answers}` → facade
  routes to the active backend (already keyed on `this.active`; Codex-active →
  `CodexBridge.handleChatQuestionReply` → `AskProxy.onQuestionReply`) → sends
  `ask.reply{askId,answers,sourcePid:serverPid}` via `InternalWsService.broadcast`.
  The matching cq-mcp broker resolves; other children ignore the unknown askId.
- No-hang on teardown: cq-mcp `rejectAll` on channel close; server `AskProxy.clear()`
  on shutdown; a late reply is stale-dropped.

## Adversarial review focus (per brief) — outcomes

1. **askId↔toolUseId↔socket correlation under reply-before-ask race** — server
   side has NO reply-before-ask race (browser cannot reply before it renders the
   synthetic event), so `AskProxy` needs no buffer. The race exists only on the
   cq-mcp side (inbound `ask.reply` vs tool-handler park); `CqMcpAskBroker`'s
   two-slot machine (per-askId `pending` + `bufferedReplies`) handles both
   orderings. Integration test ran 3× consecutively without flake.
2. **Session preempted/shut down while ask pending — both sides reject, not hang**
   — cq-mcp: `registerOnClose`→`rejectAll` rejects the parked promise (tool
   returns error). server: `AskProxy.clear()` on shutdown; late reply stale-dropped.
   Tested (`codexBridge.test.ts` post-shutdown stale case; cq-mcp `rejectAll` cases).
3. **question_reply dispatch (Claude in-process vs Codex proxy) airtight** — the
   facade routes to `this.active`; a Claude-active reply resolves the in-process
   `AskBroker` (proved by parking an ask on it and asserting resolution — would
   hang if mis-routed), a Codex-active reply becomes an `ask.reply` with the right
   askId. Both tested.

## Manual scenario — closest reproduction achieved

The brief's manual scenario (`bun run dev`, pick a Codex model, prompt the agent
to ask a question, answer in the UI, confirm the agent continues) requires real
Codex auth AND a model that reliably chooses to call `ask_user_question` — there
is no deterministic forcing function for the latter, and Codex auth is not
exercised here. The closest deterministic reproduction achieved, which covers the
entire proxy mechanism end-to-end short of the browser + live model:

- `packages/server/test/askProxy-integration.test.ts` spawns a **real cq-mcp
  subprocess** (`bun run packages/cq-mcp/src/main.ts`) connected back to a real
  `Bun.serve` + `InternalWsService` with `CQ_INTERNAL_WS_URL/TOKEN/CQ_SESSION_ID`
  set, then: (a) asserts `tools/list` includes `ask_user_question` (14 tools);
  (b) invokes the tool; (c) the server captures the inbound `ask.request` (right
  askId/toolUseId/sessionId/questions) and broadcasts a synthetic `ask.reply`
  standing in for the browser's answer; (d) asserts the tool resolves with the
  Claude-identical `CallToolResult` and `string[]`-answers normalised to `"a, b"`.

The only untested seam vs. a full manual run is the browser render + click, which
is identical to the in-process Claude ask card (same `Stream.tsx` path keyed on
the `AskUserQuestion` tool_use block — verified by `buildAskUserQuestionEvent`
unit test). A concrete E2E follow-up is recorded as `ASKPROXY-D03`.
