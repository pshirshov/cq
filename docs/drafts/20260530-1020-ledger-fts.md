## Approach (settled with user)
In-memory MiniSearch v7.2.0 index as a derived projection of `FsLedgerStore`'s
in-memory ledgers, kept coherent by the SAME hooks (`onMutation` local +
`invalidate`/`reloadLedgerFromDisk` remote) that keep the stores coherent
across processes. Not SQLite. `includeArchived` IN v1.

## Key environmental facts discovered (S4 scan)
1. MiniSearch indexes a FIXED named-field set; ledger items have heterogeneous
   per-ledger schemas. → Map each item to a doc with a CANONICAL field set
   bucketed by boost tier: `headline` (headline|title|question),
   `body` (description|rationale + remaining string/string[] fields),
   `status`. This gives stable boostable field names + cross-ledger uniformity.
2. **`createLedgerMcpTools` is consumed ONLY by the Claude in-process bridge**
   (`claudeBridge.ts:381`). The cq-mcp stdio binary
   (`cq-mcp/src/main.ts::registerLedgerTools`, ~L154-378) HAND-REDEFINES every
   tool via `server.registerTool(...)`. So `fts_search` must be added in BOTH
   places. The brief's "available on both backends automatically" holds only
   for the Claude bridge.
3. Tool-count assertions to bump: `ledger/test/mcp-tools.test.ts` (13→14),
   `cq-mcp/test/main.test.ts` (13-list → 14), `cq-mcp/test/submitRelay.e2e.test.ts`
   (14→15: "13 ledger + relay" becomes "14 ledger + relay").
4. flake.nix ledger dep symlinks are a `for dep in zod yaml unified … bun-types`
   loop (L242). Add `minisearch` to that list. cq-mcp's `@cq/ledger` is a
   symlink to the same `packages/ledger` (L263), so ONE symlink covers both
   consumers. FOD `outputHash` (L109) keys off `packages/ledger/package.json`
   + `bun.lock`; both change → refresh the hash.
5. Store-internal archived-items accessor needed: Fs reads
   `docs/archive/<ledger>/<id>.md` via existing `parseArchive`; InMemory holds
   `archives` Map keyed `<ledger>/<id>`. Both adapters already have
   `archivePointers` on each Ledger.

## Archive nuance (immutable archives)
Archive files never change once written. So:
- At `init()`: build active docs for every ledger from in-memory items, AND
  build archived docs by reading each archive pointer once.
- On `onMutation`/`invalidate` for a ledger: rebuild only the ACTIVE docs for
  that ledger (drop the ledger's active docs, re-add from current in-memory
  items). Do NOT re-read archive files.
- On `archive` op: the just-archived milestone's items move from active to
  archived. Simplest correct handling: on an `archive` mutation, re-read that
  ledger's archive pointers to (re)build its archived docs, then rebuild its
  active docs. This reads only the affected ledger's (now-immutable) archives
  on the rare archive event, not on every active edit. Active edits never touch
  archive I/O.

Implementation choice: index keyed per (ledgerId) → two doc buckets
(active, archived). A rebuild of a ledger replaces the relevant bucket(s).
Use MiniSearch `discard(docId)` for removal (id-only) + `add(doc)`; keep a
`Map<ledgerId, Set<docId>>` per bucket so per-ledger replacement is O(docs in
ledger), never a global rebuild.

## PRs (one ledger entry = one commit)

### PR-01 — `LedgerSearchIndex` + dep
- Add `minisearch@7.2.0` to `packages/ledger/package.json` deps.
- New `packages/ledger/src/search/LedgerSearchIndex.ts`:
  - Doc shape: `{ docId: "<ledgerId>:<itemId>", ledgerId, itemId, milestoneId,
    status, archived: boolean, headline, body }` + store the full `Item` for
    reconstruction (via a side `Map<docId, {ledgerId, item, archived}>` so the
    search result maps back to a real `Item`; do not rely on MiniSearch stored
    fields to rebuild the typed Item).
  - Fields indexed: `headline`, `body`, `status`. storeFields:
    `ledgerId, itemId, archived, status`. Boosts: headline 4, body 2, status 0.5.
  - Field bucketing helper: given a ledger's items, map each item's schema
    fields to headline/body using name sets
    `HEADLINE_FIELDS = {headline,title,question}`; everything else string-ish
    joins into body. status is its own field.
  - Methods (uniform async where store-touching; pure where not):
    - `rebuildLedgerActive(ledgerId, activeItems: Item[])` — pure (caller passes items)
    - `setLedgerArchived(ledgerId, archivedItems: Item[])` — pure
    - `removeLedger(ledgerId)`
    - `search(query, { ledger?, limit=20, fuzzy?, prefix?, statusFilter?, includeArchived=false })`
      → `[{ ledgerId, item: Item, score, matchedFields: string[] }]`,
      cross-ledger when `ledger` omitted, ranked desc.
  - `matchedFields` from MiniSearch `result.match` (term→fields) flattened+deduped.
  - Filter: `includeArchived=false` ⇒ `r.archived !== true`; statusFilter ⇒
    `r.status === statusFilter` (exact, case-insensitive compare).
- Acceptance: `bun test` (new unit test exercises ranking headline>body, fuzzy,
  prefix, status, includeArchived partition on a hand-built index — no store).
- Verify: `tsc -b` clean; new file lints.

### PR-02 — Wire index into stores + `ftsSearch` on `LedgerStore`
- `LedgerStore` interface: add
  `ftsSearch(query: string, opts: FtsSearchOpts): Promise<FtsSearchHit[]>`.
  Define `FtsSearchOpts` / `FtsSearchHit` types in `LedgerStore.ts`.
- Both adapters own a `LedgerSearchIndex`. Triggers:
  - Fs `init()`: for each ledger, build active from in-memory; build archived by
    reading each pointer (`fetchArchive`-style parse). Provide private
    `collectArchivedItems(ledgerId): Promise<Item[]>`.
  - Fs `fireMutation` is sync + non-throwing; index rebuild for active is sync &
    cheap (pure array map). On `op==="archive"`, ALSO refresh archived docs
    (async read) — schedule it without unwinding the write: wrap in a
    non-throwing async that logs on error. The active rebuild stays sync.
  - Fs `invalidate`/`reloadLedgerFromDisk`: after the in-memory ledger is
    replaced, rebuild that ledger's ACTIVE docs (the remote write changed active
    state; archives are immutable so no re-read needed UNLESS the remote op was
    an archive — but invalidate doesn't carry op; safe approach: on invalidate,
    rebuild active AND refresh archived for that ledger, since invalidate is
    rare and archive-refresh is idempotent over immutable files).
  - `createLedger`: index empty ledger.
- InMemory: same wiring; archived items from its `archives` Map (group items)
  for that ledgerId. `invalidate` is a no-op there (no external source) — index
  already current from the local mutation path.
- onMutation rebuild MUST NOT throw into the write path: the active rebuild is
  pure (cannot throw on valid items); still guard the archive-refresh async.
- Acceptance: dual-tests in `store-abstract.ts` cover ftsSearch ranked
  cross-ledger; single-ledger filter; boosts; fuzzy/prefix; status; includeArchived
  (archive a milestone then assert found only when include_archived=true);
  local coherence (create/update reflected; archive moves active→archived).
- Verify: `bun test` (both adapters green).

## Risks / assumptions
- R1: MiniSearch `discard` leaves tombstones; per-ledger rebuild on every edit
  could accumulate dirt. Mitigation: autoVacuum default on; small N. Acceptable.
- R2: Item with no headline-bucket field (e.g. goals has `description` required,
  no headline) → headline empty, body carries it. Fine; boosts still order when
  a headline exists.
- R3: cross-ledger global item-id uniqueness holds, but docId keyed by
  ledger+item is safe regardless.
- R4: e2e for fts_search — mirror ledger-create.spec.ts only if cheap; else
  defer with a note (brief says optional/preferred).
- A1: status filter is exact-match (not substring) — matches `search_items`
  semantics being substring is separate; fts status is a precise filter.
