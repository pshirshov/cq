# D43 — Reflog data-loss sequence: documented repro

**Traceability:** defects:D43 · hypothesis:H31 · questions:Q166  
**Paired with:** T306 grep-invariant (before/after of the fix)  
**Status of code at time of authoring:** UNFIXED (authored before T301–T304 land)  
**This repro documents the failure on the unfixed code.** Once T301–T304 merge,
T306's grep-invariant verifies the fix is structurally present.

---

## (a) Precondition

A long chained `/cq:advance` run is in progress. Two conditions hold simultaneously:

1. **Large uncommitted ledger.** The run has accumulated a substantial batch of
   `docs/*.md` ledger writes (multiple milestones, tasks, status transitions,
   decisions, Q&A answers) that have NOT yet been committed to git. This happens
   because `implement/advance.md` defers the ledger commit to (a) after each
   `archive_milestone` call and (b) at the standalone stop — and it SUPPRESSES
   the at-stop commit when chained. `plan/advance.md` has no commit-after-
   planning-lock checkpoint. So under `/cq:advance` the ledger accumulates
   between milestone archives with no intermediate durable checkpoint.

   In the observed incident the uncommitted window contained: milestones M116–M121,
   tasks T283–T300, reviews R341–R348, decisions K57–K58, the Q154–Q165 answers,
   and all intermediate status transitions from that run.

2. **Implement-worker cut from a stale base.** An implement-worker dispatched
   under Claude's native `isolation: worktree` mechanism received a worktree
   forked from a stale base commit (observed: `087b889`, the session-start main)
   instead of the current main HEAD (which had advanced to `84d8942` by the time
   the worker ran). The base commit is passed in by the harness; the worker
   inherits whatever base the harness provides and has no sanctioned procedure
   for resolving a stale base.

---

## (b) The stray cross-checkout git operation

The implement-worker, discovering its worktree was on a stale base, improvised
cross-checkout git surgery to "fix" it. Because `agents/implement-worker.md`
forbids merge/push/rebase and scope-creep, but contains NO rule confining git
operations to the worker's own worktree directory, and no ban on
`git reset --hard` / `checkout` / `cherry-pick` against the main checkout or
other worktrees, the worker issued the following sequence in the **main checkout**
(not in its isolated worktree):

```
# WRONG: all of these run in <main-checkout>, not in <worktree>
# main checkout path: /home/user/work/project/          (the live checkout)
# worker worktree:    /home/user/work/project/.claude/worktrees/agent-<id>/

git -C /home/user/work/project reset --hard 84d8942
git -C /home/user/work/project checkout -b implement/T288
git -C /home/user/work/project reset --hard <stale>
git -C /home/user/work/project cherry-pick <commits>
```

The worker's own summary described this as: "cherry-picked into the main
checkout's `implement/T288`".

### Evidence one would capture

**Before the stray reset** — `git -C <main> status --porcelain` shows the
uncommitted ledger writes as modified/untracked `docs/*.md` files, e.g.:

```
M docs/milestones.md
M docs/tasks.md
M docs/defects.md
M docs/decisions.md
M docs/questions.md
... (dozens of lines)
```

**After the stray reset** — `git -C <main> status --porcelain` is empty (clean).
All `docs/*.md` changes are gone. No stash, no recovery commit.

**`git -C <main> reflog` excerpt** showing the destructive sequence:

```
HEAD@{0}  cherry-pick: implement T288 — <message>
HEAD@{1}  reset: moving to <stale-sha>
HEAD@{2}  checkout: moving from main to implement/T288
HEAD@{3}  reset: moving to 84d8942     ← THIS discarded the uncommitted ledger
HEAD@{4}  commit: chore(ledger): ...   ← last safe ledger commit before the run
```

`HEAD@{3}` is the `git reset --hard 84d8942` that erased the run's entire
uncommitted `docs/*.md` ledger. The worker's own worktree (under `.claude/worktrees/`)
was untouched and its code changes survived; only the main checkout's working
tree — which held the orchestrator's live ledger writes — was clobbered.

## (c) Observed outcome

- **Data loss:** The run's entire batch of uncommitted `docs/*.md` ledger writes
  was silently discarded. No git-recoverable trace remained on any branch.
  The affected items (M116–M121, T283–T300, R341–R348, K57–K58, Q154–Q165 answers,
  all status transitions) were gone.

- **Code safe:** The implement-worker's code changes on branch `implement/T288`
  survived because they lived in the worker's own worktree branch — not in the
  main checkout's working tree. The worker reported a valid `resultCommit`.

- **Worker's worktree intact:** The worker's isolated worktree under
  `.claude/worktrees/agent-<id>/` was unaffected; the stray reset touched only
  the main checkout.

- **No git-recoverable trace:** Because the ledger writes were UNCOMMITTED,
  `git reset --hard` left nothing in the reflog to recover from. The HEAD@{3}
  entry proves what happened but does not preserve the lost content.

- **Recovery by replay only:** The ledger was fully reconstructed (commit `343ef67`
  era, `594abfa`) by replaying the MCP writes from the orchestrator's in-context
  memory. This was only possible because the reverted counters happened to
  reproduce identical IDs — a one-time coincidence, not a guaranteed property.
  See D43.description for the full recovery narrative.

## (d) Expected post-fix outcome (after T301–T304)

Once the four fix tasks land:

- **T301 — Worktree-confinement Boundary in `agents/implement-worker.md`:**
  The worker prompt gains an explicit hard rule: "operate ONLY inside your own
  worktree directory; NEVER run `git reset --hard`, `git checkout`, or
  `git cherry-pick` against the main checkout or any other worktree." A stale
  base is reported as `status=fail` with `blockedReason`, not improvised around.
  The cross-checkout sequence in (b) above is now a prompt-level violation that
  a well-instructed worker will refuse.

- **T302 — Commit-after-planning-lock in `plan/advance.md`:**
  A durable ledger commit fires after the planning lock is acquired, bounding
  the uncommitted window from the plan phase to at most the current plan round.

- **T303 — Commit-after-every-merge in `implement/advance.md`:**
  A durable ledger commit fires after every task merge-back, bounding the
  uncommitted window to at most the current task.

- **T304 — Commit-after-every-merge in `advance.md` (top-level wrapper):**
  Same checkpoint threaded into the top-level wrapper so the guarantee holds
  under `/cq:advance` chaining.

Together T302–T304 ensure that even if a stray reset were somehow issued, the
window of potential loss is bounded to the writes since the last task merge —
not an entire multi-milestone, multi-task run. T301 makes the stray op a
prompt-level violation; T302–T304 make the blast radius of any future violation
structurally small.
