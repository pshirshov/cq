## Reproduction (red-before-green)

Wrote `packages/server/test/warmup-activity.test.ts` asserting a freshly-started

## Root cause (confirmed by code inspection)

`ClaudeBridge` constructed `ActiveSession` with `turnInFlight: true`
(claudeBridge.ts:647). The old comment claimed the first turn begins on session
creation, but `handleChatStart` seeds an EMPTY input queue (`new AsyncQueue` at
:432) and delivers no prompt; the first real turn arrives via `chat.input` →
`setTurnInFlight(true)` (:849) and is cleared by the per-turn `chat.done` in
`sendDone` (:1517). So a fresh session reported in-flight with nothing streaming,
and the warm-up session (never runs a real turn) never reached `sendDone` →
`running=1` latched → badge stuck "BUSY (1)".

CodexBridge was already correct: `isTurnInFlight()` ⟺ `abortController !== null`,
constructed `null` (codexBridge.ts:604). No change needed on that backend.

## Fix

- `claudeBridge.ts:651`: `turnInFlight: false` at construction + corrected comment.
- No change to the turn lifecycle, tracker aggregation, workflow dispatch, pool=1.
- No window where a real turn streams while the flag is false: `setTurnInFlight(true)`
  runs synchronously in the same `handleChatInput` call that enqueues the input, and
  the SDK emits `result`/`chat.done` only after receiving that input.
