## Goal

App-wide dark theme: light / dark / auto (default auto = follow OS), toggle in
the gear popup, applied live + persisted. `[data-theme="dark"]` on `<html>`
overrides CSS variables; `auto` resolved in JS via `matchMedia` with a live OS
listener. The ~28 CSS modules that hardcode light hexes get swept onto the
variables so dark applies everywhere; light stays byte-for-byte identical in
resolved color.

## Variable taxonomy (global.css)

`:root` keeps the existing LIGHT values (UNCHANGED) and gains the new semantic
variables below with light values. `[data-theme="dark"]` overrides every one
with a dark value. Modules reference `var(--name)` (the existing
`var(--name, #fallback)` fallback form is retained where present — fallbacks
are inert once the var is defined, and proving light-unchanged uses the
resolved value, not the fallback).

Existing base (light, unchanged): `--bg #ffffff`, `--surface #fafafa`,
`--surface-message #f7f7f8`, `--border #e0e0e0`, `--border-light #f0f0f0`,
`--text-primary #111111`, `--text-secondary #666666`, `--accent #0070f3`,
`--hover #f0f0f0`, `--input-bg #ffffff`, `--status-error #991b1b`.

- `--text-tertiary`          #888888 → #8a8a8a   (the #888/#999 muted labels)
- `--on-accent`              #ffffff → #ffffff    (text on accent/colored buttons)
- `--shadow`                 rgba(0,0,0,0.15) → rgba(0,0,0,0.5)
- `--shadow-strong`          rgba(0,0,0,0.25) → rgba(0,0,0,0.6)

Syntax-highlight colors (Thinking/Markdown/CodeBlock inline tokens, VS-Code /
Catppuccin literals): these are token colors inside code/markdown that are
intentionally a fixed editor palette, NOT theme surfaces. Treat as in-scope
ONLY where they paint a card surface/border/text (e.g. Thinking `.root`
border/bg, ElicitationCard chrome). Pure syntax token literals that already
read on both a light and dark code background may stay as documented
exceptions (listed per-file in the sweep). Shiki itself emits its own theme.

## Milestones / PRs

- **dark-1** — global.css dark variable set + `theme.ts` controller.
  - global.css: add new semantic vars to `:root` (light); add full
    `[data-theme="dark"]` block overriding every theme var. Light `:root`
    values UNCHANGED.
  - `lib/theme.ts`: `ThemeMode = 'light'|'dark'|'auto'`; `readMode()`/
    `writeMode()` (localStorage `cq.theme`, validated, try/catch like ChatTab's
    ls helpers); `resolveMode(mode, mql)` → `'light'|'dark'`; `applyMode()`
    sets `document.documentElement.dataset.theme`; `initTheme()` reads +
    applies + attaches a `matchMedia('(prefers-color-scheme: dark)')` change
    listener that re-resolves when mode is auto; returns a controller object
    `{ getMode, setMode, dispose }`. No sync/async unions (all synchronous —
    no promises needed; DOM + localStorage are sync).
  - main.tsx: call `initTheme()` before `createRoot(...).render` so the
    attribute is set before first paint (no flash). Keep the controller in a
    module-level singleton for SettingsPopup to reach via a small accessor.
  - Acceptance: check stays 1059/0 (no test yet); manual: dataset.theme set.

- **dark-2** — Theme control in SettingsPopup, applied live.
  - SettingsPopup gains a Theme `<select>` (Light/Dark/Auto), value read from
    the controller (`getMode()`), `onChange` → `controller.setMode(v)` which
    persists + applies immediately. Wired through Header → ChatTab like the
    other gear settings BUT the value does NOT enter ChatStart/session state —
    it is display-only (localStorage + DOM via the controller). Minimal
    threading: SettingsPopup takes `themeMode` + `onThemeModeChange` props;
    Header passes them through; ChatTab owns `themeMode` state seeded from the
    controller and its onChange calls `controller.setMode`. Hint text stays
    "Changes apply to the next new chat" for the per-session rows; the Theme
    row is visually separated and not governed by that hint (it is live).
  - Acceptance: settingsPopup.test additions green; existing tests green.

- **dark-3** — the CSS sweep (~28 modules).
  - Replace every hardcoded theme-sensitive hex/rgb (background, color,
    border, box-shadow, fill) with the appropriate var. Keep `var(--x, #fb)`
    fallback form where already used; for newly-introduced vars use the same
    form with the prior literal as the fallback. Status/severity chips → the
    `--chip-*` pairs. Code chrome → `--surface-code`/`--border-code`/etc.
    Solid colored buttons → `--surface-success`/`--surface-danger`/
    `--surface-neutral` + `--on-accent`. Document per-file any intentional
    exception (pure syntax token literals).
  - Acceptance: the sweep assertion test (dark-4) passes; light resolved
    colors unchanged (computed-style diff in a unit test + manual).

- **dark-4** — tests + e2e + discharge.
  - `theme.test.ts`: persist/read `cq.theme`; auto resolves via injected
    matchMedia (dark when OS dark, light otherwise); sets dataset.theme;
    updates on OS-change event; explicit light/dark override OS.
  - settingsPopup theme additions (in dark-2 file or here): renders
    Light/Dark/Auto, persists choice, applies it (asserts dataset.theme).
  - sweep assertion test `dark-sweep.test.ts`: scans the theme-sensitive
    modules; asserts no bare known-light surface/text/border hex on
    theme-sensitive props (must be `var(--…)`); whitelists documented

  - light-unchanged test: a unit test that resolves a sample of variables
    under default (light) and asserts they equal the original `:root` hexes.
  - e2e `dark-theme.spec.ts` (main project): open → gear → Theme=Dark →
    assert `<html data-theme="dark">` + a sample surface computed

  - Discharge: `bun run check` exit 0 ×2; `bun run e2e` green; `nix build

## Risks / assumptions

- Flash-of-wrong-theme: `initTheme()` runs in main.tsx before render, but the
  CSS link + JS bundle load after the empty index.html. Acceptable — the page
  is blank (var(--bg)) until JS runs; dataset.theme is set synchronously in the
  same tick the bundle executes, before React paints. No inline head script is
  added (would require editing the server's buildWeb HTML emit — borderline
  WEB-ONLY; deferred unless flash is observed). Documented.
- Light-unchanged proof: assert resolved (computed) values, not source text.
- Dark-applies-app-wide proof: sweep assertion (no module hardcodes a light
  surface) + e2e computed-background check on a real surface.
- Chip legibility: each chip has an explicit dark bg+text pair chosen for
  contrast on the dark surface.
