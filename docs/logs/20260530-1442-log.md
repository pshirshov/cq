## Original request

Implement a real, app-wide dark theme for cq's web app: light / dark / auto
(default auto = follow OS), toggle in the gear popup, applied live + persisted.
Mechanism: `data-theme` on `<html>`, `[data-theme="dark"]` CSS overrides, auto
resolved in JS via matchMedia with a live OS listener. Plus a CSS sweep of the
~28 `styles/*.css` modules that hardcode light hexes, so dark applies
everywhere while light stays visually unchanged.

## How the two adversarial invariants were proven

- **Light unchanged (visually neutral sweep):** because every theme var is
  defined in `:root`, a module's `var(--X, #fb)` fallback is INERT — the
  resolved light color is `:root[--X]`. The sweep wrote each fallback to the
  EXACT prior literal, and a committed test (`darkSweep.test.ts`) asserts
  `#fb == :root[--X]` for every such usage (modulo 9 pre-existing inert
  baseline fallbacks). So resolved-light == original literal everywhere the
  sweep touched. A scripted git-diff audit independently confirmed 0
  light-shift on branch-changed lines.
- **Dark applies app-wide:** 118-var light/dark PARITY test (no var left at its
  light value in dark); `buildWeb()` emits a `main.css` containing
  `[data-theme="dark"]` + dark `--bg:#121212`; and the e2e samples a REAL
  surface's computed `background-color` (luminance < 80 dark, > 200 light) —
  proving the sweep, not just the attribute.

## Exceptions (intentional, documented)

Always-dark chrome left literal because theming it would change the LIGHT
appearance: Header bar, Tooltip panel, Toast stack, SettingsPopup chrome,
Indicator state-dots (semantic). SearchBar + Stream were already
variable-driven. Within swept files: History `.resumeButton` (dark button on
a light table) and the ElicitationCard translucent focus glow. Shiki code
bodies render `github-dark` always (pre-existing, CodeBlock.tsx) — only their
container chrome was swept.

## Notes / caveats for future work

- Auto/OS-dark live-resolution is covered by the unit test via a matchMedia
  mock — Playwright cannot easily drive the live OS scheme through the same
  code path, so the e2e covers explicit Light/Dark only. Documented in the
  spec + plan.
- One transient cross-file DOM-pollution flake (attachments Case 4) appeared
  once during a mid-cycle full run; it passed on re-run and in isolation —
  pre-existing nondeterministic bun-test ordering (see `test/helpers/dom.ts`
  docs), not caused by this change. The two final `bun run check` runs were
  both clean 1077/0.
- No-flash-on-load: `initTheme()` runs before `createRoot().render`; no inline
  head script was added (would require editing the server's `buildWeb` HTML
  emit — borderline WEB-ONLY; deferred unless a flash is observed). The page is
  blank (`var(--bg)`) until the JS bundle executes, which sets `data-theme`
  synchronously before React paints.
